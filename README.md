## How this works

This repository is the status infrastructure itself — powered by [Upptime](https://github.com/upptime/upptime), with no server anywhere:

- **GitHub Actions** checks every monitored endpoint every 5 minutes and records response times as git commits — the uptime history in [`history/`](./history) is append-only and auditable.
- **GitHub Issues** are the incident log: when an endpoint goes down, an issue opens automatically with a timeline, and closes when it recovers. Every incident this ecosystem has ever had is public in [Issues](https://github.com/usetheodev/theo-status/issues).
- **GitHub Pages** serves the status site at [status.usetheo.dev](https://status.usetheo.dev).

By design, none of this runs on the infrastructure it monitors. If usetheo.dev or TheoCloud have a bad day, this page does not go down with them.

## Why it's public

Part of Theo's [honesty policy](https://usetheo.dev/honesty): uptime claims should be backed by accumulated, verifiable history — not by a badge. This repo *is* that history, from day one (2026-07-07). If a check fails, it lands here too.

## Configuration

Everything lives in [`.upptimerc.yml`](./.upptimerc.yml): the monitored endpoints, the status site branding, and the custom domain. To add an endpoint, add a `sites:` entry and push — the next cron cycle picks it up. Full options: [Upptime docs](https://upptime.js.org/docs/configuration).

Operational notes:

- `get.usetheo.dev` is intentionally not monitored yet — the domain doesn't resolve today. Add it the day it ships; a monitor born red is noise, not signal.
- DNS for `status.usetheo.dev` is a Cloudflare-proxied CNAME to `usetheodev.github.io`; HTTPS is served by the Cloudflare edge (zone SSL mode `full`). Don't unproxy the record without a plan — GitHub Pages' own certificate was never issued.
- Workflow files under `.github/` are synced from the upstream Upptime template (see `.templaterc.json`) — customize behavior via `.upptimerc.yml`, not by editing workflows.
